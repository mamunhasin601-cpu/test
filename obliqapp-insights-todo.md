# TODO по итогам разбора obliqapp.ru — ревизия v2

Источник: скринкаст конкурента obliqapp.ru + решения по архитектуре после обсуждения.
Ключевые решения ревизии:
- Фото товара — **обязательное** поле. Сценарий «только по описанию» убираем.
- Основной движок генерации — **`google/gemini-3.1-flash-image` (Nano Banana 2)** как img2img: держит identity товара (цвет, принт, фактура, логотип) при переносе в новую сцену.
- **`openai/gpt-image-2`** оставляем только для инфографики (кириллица поверх фото).

---

## P0 — критично, делаем в первую очередь

### 1. Обязательное фото товара + приватный бакет
- На форме `dashboard.index.tsx` убрать возможность отправить запрос без изображения. Валидация на клиенте + на сервере (`InputSchema` в `ai.functions.ts`).
- Использовать существующий бакет `uploads` (уже есть, приватный) для исходных фото пользователя. Отдельно от `generations`.
- Путь: `${userId}/inputs/${uuid}.${ext}`.
- Для передачи в Nano Banana — signed URL на 1 час (Gemini скачивает по URL сам).
- В аналитике `analyzeProduct` убрать ветку fallback «нет изображения — анализ по описанию»: раз фото обязательно, ветка мертвая.

### 2. Nano Banana как img2img движок для всех фото-кадров
Заменить текущий `openai/gpt-image-2` в `generateCard` на **`google/gemini-3.1-flash-image`** для всех сценариев, кроме инфографики.

Технически:
- Тело запроса — chat-shape (см. `ai-multimodal-input`): `messages` + `modalities: ["image", "text"]`, во входе — `image_url` с signed URL исходного фото + `text` с инструкцией сцены.
- Стриминг `stream: true` — партиалы приходят по мере рендера, показываем прогрессивно с blur.
- Endpoint остаётся `https://ai.gateway.lovable.dev/v1/images/generations`, но тело переписать полностью — не переносить `size`/`quality`/`partial_images` от gpt-image-2.
- Ответ нормализован к OpenAI-shape (`b64_json`) — парсер тот же.

### 3. Выбор типа кадра после анализа (6 сценариев)
После анализа gpt-5-mini показать сетку карточек:
- **Каталог** — packshot на нейтральном/белом фоне, без модели.
- **На модели** — студийный кадр, человек носит товар (для apparel/аксессуаров, без головы согласно правилам WB).
- **Как в магазине** — товар на вешалке/манекене в интерьере.
- **Раскладка (flat-lay)** — вид сверху, товар + сопутствующие вещи.
- **В жизни (lifestyle)** — реальная сцена использования.
- **Крупный план** — деталь: шов, фактура, застёжка, логотип.

Для каждого — свой prompt-билдер поверх текущего `PRESETS[marketplace]`. Nano Banana получает исходное фото + сцену и переносит товар в неё.

Прятать несовместимые варианты (например, «На модели» скрывать для не-apparel категорий).

### 4. Двухступенчатая генерация с чек-листом
Разбить `generateCard` на два серверных вызова:
- **`analyzeAndPlan`** — принимает фото + описание + категорию + marketplace + shotType. Возвращает: `analysis`, предлагаемые буллеты текста (для WB), финальный prompt, оценку стоимости в кредитах. **Кредиты НЕ списывает.**
- **`confirmAndGenerate`** — принимает результат `analyzeAndPlan` + подтверждение пользователя. Списывает кредиты, вызывает Nano Banana, кладёт в `generations`, возвращает signed URL.

Между шагами — UI-подтверждение:
```
Товар: Черный спортивный костюм
Кадр: На модели
Текст на карточке — проверьте:
  • Черный спортивный костюм
  • Футболка с коротким рукавом
  • Брюки зауженные к низу
[Подтверждаю, генерируй  ✦ 8]  [Изменить]
```

## P1 — важное, сразу после P0

### 5. Инфографика — отдельный shotType на gpt-image-2
- Единственный сценарий, где остаётся `openai/gpt-image-2` — у него кириллица стабильнее.
- Тело: `prompt` + `image_url` input (не text-only!), чтобы текст ложился поверх реального фото товара, а не поверх перерисовки.
- CLOSED LIST зон размещения текста (как у obliqapp): top-left / top-center / top-right / bottom-* / вертикальная колонка / внутри пустой негативной зоны.
- Буллеты подтверждаются на шаге `analyzeAndPlan` (как в п.4).

### 6. Действия после результата
Под сгенерированной картинкой:
- **Скачать** (есть)
- **Другой вариант** — перегенерировать тем же prompt-ом
- **Правка в чат** — свободный текст, применяется как diff к prompt-у, запускает новую Nano Banana с тем же исходником
- **Сделать следующий слайд** — быстрый переход к выбору другого shotType для того же товара (тот же исходник, тот же analysis, экономим кредиты)

### 7. Множественные варианты за один запуск
Опция `n = 1 / 2 / 3` с ценой ×N. Цикл вокруг вызова Nano Banana (не поддерживает `n` нативно в chat-shape), запускать параллельно `Promise.all`.

### 8. Стоимость в кредитах у каждого шага
Показывать рядом с кнопками: анализ ~2, генерация Nano Banana ~8, инфографика gpt-image-2 ~12, доп. вариант ×N. Не одна цифра «15» после нажатия.

## P2 — полезные улучшения

### 9. Человеческая сводка вместо сырого промпта
По умолчанию: «Стиль: студийный · Свет: мягкий · Фон: белый · Текст: 3 буллета · Формат: 3:4».
Кнопка «Показать техзадание» — раскрывает полный prompt для тех, кому нужно.

### 10. Правка в чат внутри карточки результата
Поле «напишите правку» под результатом — свободный текст применяется как diff к prompt-у и запускает новую генерацию Nano Banana с тем же исходным фото.

### 11. Мультислайдовая карточка товара
Хранить сгенерированные слайды под одной сущностью «товар», а не как разрозненные `generations`.
- Новая таблица `products` (user_id, name, source_image_path, analysis, category, created_at).
- В `generations` добавить `product_id` FK и `shot_type`.
- UI: «Товар: Черный спортивный костюм — 4/10 слайдов готово», кнопки быстрого запуска недостающих типов.

## Что НЕ делаем

- Не тащим полностью тёмный чат-only UI без сайдбара — наш dashboard понятнее.
- Не показываем 9k символов английского промпта по умолчанию — оставить скрытым за «Показать техзадание».
- Не отказываемся от пресетов маркетплейсов — жёсткие правила WB/Ozon/ЯМ надёжнее для модерации.
- Не поддерживаем сценарий «сгенерируй по одному описанию без фото» — упрощаем архитектуру.
- Не используем `gpt-image-2` для основной генерации сцен — на входном фото Nano Banana лучше сохраняет identity товара.

---

## Что бережём (не сломать при рефакторинге)

- Пресеты маркетплейсов (`wb / ozon / ym`) с правилами формата, текста, фона — работают поверх любого движка.
- Двухступенчатая аналитика (сначала анализ, кредиты только если анализ ок) — расширяем, но не выбрасываем.
- Русскоязычный `ANALYST_SYSTEM_PROMPT` (gpt-5-mini) с алгоритмом идентификации по фото + официальным характеристикам.
- Dashboard с историей, биллингом и настройками.

## Технические заметки

- **Chat-shape Nano Banana body** (см. `ai-multimodal-input`):
  ```json
  {
    "model": "google/gemini-3.1-flash-image",
    "messages": [{ "role": "user", "content": [
      { "type": "text", "text": "<инструкция сцены + правила маркетплейса>" },
      { "type": "image_url", "image_url": { "url": "<signed URL фото товара>" } }
    ]}],
    "modalities": ["image", "text"],
    "stream": true
  }
  ```
  Не переносить `size`, `quality`, `partial_images`, `n` — не поддерживаются.

- **Стриминг**: с `stream: true` идут `image_generation.partial_image` + финальный `image_generation.completed`. UI показывает партиалы с CSS blur, снимает blur на completed. См. `ai-image-generation` для парсера с `flushSync` (обязательно, иначе React 18 схлопывает партиалы).

- **Signed URL для фото**: 1 час хватает с запасом на аналитику + генерацию. Не делать вечные URL.

- **Ошибки Nano Banana**: 429 → rate limit, 402 → credits exhausted (для LOVABLE_API_KEY), 400 с `content_policy` → предложить другое фото. Не молчать.
